# .gitignore Configuration Guide

This document explains the .gitignore configuration for TruAxisVentures project.

## Overview

The project has two .gitignore files:
- **Root `.gitignore`**: For frontend/Node.js files
- **Backend `.gitignore`**: For Python/Flask files

## What's Being Ignored

### üîí Security & Sensitive Data (NEVER COMMIT)
- `.env` files (contains database passwords, API keys)
- `*.pem`, `*.key`, `*.cert` (SSL certificates and private keys)
- `secrets/`, `credentials/` directories
- Database files (`*.db`, `*.sqlite3`)

### üì¶ Dependencies (Can be reinstalled)
- `node_modules/` (frontend dependencies)
- `venv/`, `env/` (Python virtual environments)
- `__pycache__/` (Python compiled files)

### üèóÔ∏è Build Output (Generated files)
- `dist/` (frontend build output)
- `build/` (build artifacts)
- `*.pyc`, `*.pyo` (Python compiled files)

### üìù Logs (Runtime generated)
- `logs/` directory
- `*.log` files
- Debug logs

### üìÅ Uploads (User generated content)
- `uploads/*` (uploaded images/files)
- `.gitkeep` files are kept to preserve folder structure

### üíª IDE & Editor Files
- `.vscode/` (VSCode settings)
- `.idea/` (PyCharm/IntelliJ settings)
- `*.swp`, `*.swo` (Vim temporary files)

### üñ•Ô∏è Operating System Files
- `.DS_Store` (macOS)
- `Thumbs.db` (Windows)
- `*~` (Linux backup files)

## What's NOT Ignored (Should be committed)

### ‚úÖ Source Code
- All `.js`, `.jsx`, `.py` files
- All `.html`, `.css` files
- Configuration templates

### ‚úÖ Configuration Templates
- `.env.example` (template without sensitive data)
- `config.example.py`

### ‚úÖ Documentation
- `README.md`
- `*.md` files
- Documentation files

### ‚úÖ Static Assets
- Images in `public/` (product images, logos)
- Fonts, icons
- Favicon

### ‚úÖ Database Schema
- Migration files
- Schema definitions
- `models.py`

## Important Notes

### 1. Environment Files
- ‚úÖ `.env.example` is committed (template)
- ‚ùå `.env` is ignored (contains secrets)
- Always update `.env.example` when adding new variables

### 2. Uploads Directory
- The `uploads/` folder structure is preserved with `.gitkeep`
- Actual uploaded files are ignored
- In production, use cloud storage (S3, Cloudinary)

### 3. Database Files
- Development database files (`.db`, `.sqlite3`) are ignored
- Production uses external database (Hostinger MySQL)
- Never commit database files with user data

### 4. Build Files
- `dist/` is ignored (generated by `npm run build`)
- Always rebuild on deployment
- Don't commit build artifacts

## Checking What's Ignored

To see what files are being ignored:
```bash
git status --ignored
```

To check if a specific file is ignored:
```bash
git check-ignore -v filename
```

## Force Adding Ignored Files (Use Carefully)

If you need to commit an ignored file:
```bash
git add -f filename
```

‚ö†Ô∏è **Warning**: Only do this if you're absolutely sure the file doesn't contain sensitive data!

## Common Mistakes to Avoid

### ‚ùå DON'T Commit:
- `.env` files with passwords
- `node_modules/` directory
- `venv/` or `env/` directories
- Database files with user data
- API keys or secrets
- SSL certificates or private keys

### ‚úÖ DO Commit:
- `.env.example` (template)
- Source code
- Documentation
- Configuration templates
- Static assets (logos, images)

## Cleaning Up Already Committed Files

If you accidentally committed ignored files:

```bash
# Remove from git but keep locally
git rm --cached filename

# Remove directory from git but keep locally
git rm -r --cached directory/

# Commit the removal
git commit -m "Remove ignored files"
```

## Best Practices

1. **Always check before committing**:
   ```bash
   git status
   git diff
   ```

2. **Review .env.example regularly**:
   - Keep it updated with new variables
   - Remove sensitive default values

3. **Use environment-specific files**:
   - `.env.development`
   - `.env.production`
   - All should be ignored except `.example` files

4. **Document sensitive variables**:
   - Add comments in `.env.example`
   - Explain what each variable does

5. **Regular audits**:
   - Check what's being committed
   - Review .gitignore periodically
   - Update as project grows

## Security Checklist

Before pushing to Git:
- [ ] No `.env` files
- [ ] No database files
- [ ] No API keys or passwords
- [ ] No SSL certificates
- [ ] No user uploaded content
- [ ] No compiled/build files
- [ ] `.env.example` is updated

## Questions?

If you're unsure whether to commit a file:
1. Does it contain sensitive data? ‚Üí Don't commit
2. Can it be regenerated? ‚Üí Don't commit
3. Is it user-generated content? ‚Üí Don't commit
4. Is it source code or documentation? ‚Üí Commit

When in doubt, don't commit!
